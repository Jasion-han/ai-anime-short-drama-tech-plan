# 技术栈方案

更新日期：2026-03-18

## 1. 目标

本文件定义内部生产系统首期推荐技术栈，用于支撑数据库主事实源、异步任务执行、镜像同步和内部控制台。

## 2. 首期技术选型

### 2.1 后端

- `Python`
- `FastAPI`
- `Pydantic`
- `SQLAlchemy`
- `Alembic`

选择原因：

- 适合 AI 工具接入和工作流编排
- 文本、图片、视频、TTS 适配器实现成本低
- 与现有 AI 执行和脚本生态兼容

### 2.2 数据层

- `PostgreSQL`
- `Redis`
- `S3` 兼容对象存储

选择原因：

- PostgreSQL 适合事务、审批、审计和多项目查询
- Redis 适合队列、缓存和短期协调
- 对象存储适合视频、音频、图像等大文件

### 2.3 异步执行层

- `Celery` 或 `RQ`

选择原因：

- 适合供应商调用重试
- 适合镜头级任务分发
- 适合镜像同步和预算检查等后台任务

### 2.4 前端

首期建议：

- 简单内部控制台
- 可选 `Next.js`

要求：

- 先做项目管理、任务队列、审批、预算、镜头列表
- 不优先做复杂创作界面

### 2.5 开发与部署

- `Docker Compose` 作为首期本地和测试环境启动方式
- `pytest` 作为测试框架
- `ruff` 或同类工具做静态检查

## 3. 运行单元

首期建议拆成以下运行单元：

- `api-server`
- `workflow-worker`
- `tool-workers`
- `mirror-sync-worker`
- `postgres`
- `redis`
- `object-storage`

## 4. 服务职责

### `api-server`

负责：

- 项目接口
- 脚本与镜头接口
- 审批接口
- 查询接口
- 状态判定接口

### `workflow-worker`

负责：

- 阶段自动推进
- 预算阈值判断
- 任务生成
- fallback 触发

### `tool-workers`

负责：

- 文本生成
- 图片生成
- 视频生成
- TTS
- QC

### `mirror-sync-worker`

负责：

- 数据库导出到 `project/`
- 生成 Markdown / CSV 镜像

## 5. 依赖边界

- 业务服务不直接耦合具体供应商 SDK
- 所有供应商调用都通过 adapter 层
- 镜像导出不反向写数据库，除非显式导入流程存在

## 6. 首期不建议采用的栈

- 细粒度微服务
- Kubernetes-first 部署
- 事件总线过度设计
- 高复杂度低代码工作流平台

原因：

- 当前阶段主要风险在流程和治理，不在基础设施规模
- 首期应先把编排和规则跑顺

## 7. 扩展路线

当项目量和并发量上升后，再考虑：

- API 与 worker 横向扩容
- 对象存储冷热分层
- 更细的任务优先级队列
- 事件驱动消息总线
- 更完整的前端控制台
