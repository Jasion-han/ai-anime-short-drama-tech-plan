# 运维、安全与合规方案

更新日期：2026-03-18

## 1. 目标

本方案用于定义内部生产系统的首期运维、安全、审计与合规底线，确保系统在多项目并行和多人协作下仍然可控、可追、可恢复。

## 2. 环境策略

至少区分以下环境：

- `dev`
- `staging`
- `prod`

要求：

- 测试环境与生产环境密钥分离
- 测试环境生成内容不得默认视为可商用产物
- 生产环境审批流和预算卡点必须生效

## 3. 密钥与凭证

- 所有第三方平台密钥统一存放在密钥管理系统
- 不允许出现在 Git 仓库、CSV、Markdown、日志明文中
- Worker 通过受控方式读取凭证
- 高成本平台建议按项目或工作空间拆分额度

## 4. 权限与审计边界

角色、审批、人工覆盖、审计字段和留痕规则，统一以 `permissions_and_audit.md` 为准。

本文件只保留与运维和安全直接相关的控制要求：

- 权限系统必须在服务端执行
- 审计日志必须可长期保留
- 高风险管理动作必须可追溯

## 5. 数据与产物存储

结构化数据：

- 存放于 `PostgreSQL`

产物数据：

- 存放于 `S3` 兼容对象存储

仓库镜像：

- 仅用于项目可读镜像和协作审阅

要求：

- 产物路径必须可追踪到 `run_id`
- 删除策略必须受控
- 重要交付物需要版本保留

## 6. 监控与告警

至少监控以下内容：

- 队列积压
- Worker 失败率
- 外部工具调用超时率
- 项目预算消耗
- 审批积压
- 镜像同步失败

建议设置告警：

- 单项目预算超过 `80%`
- 关键工具连续失败
- 项目卡在某阶段超过阈值
- 关键审批超时未处理

## 7. 异常分级与处置

异常分为：

- `P1`：生产阻塞或高风险合规问题
- `P2`：阶段推进受阻或关键工具不可用
- `P3`：局部任务失败，可回退
- `P4`：非关键告警或体验问题

处置原则：

- `P1` 必须立即暂停相关项目推进
- `P2` 允许走 fallback，但需记录
- `P3` 可自动重试或重新排队
- `P4` 进入排期处理

## 8. 合规控制点

至少在以下节点执行检查：

- 资产进入正式生产前
- 新供应商进入正式链路前
- 导出前

检查范围包括：

- 素材来源与版权归属
- 真人声音或肖像授权
- 品牌元素使用
- 输出条款与可商用性
- 平台 AI 披露要求

## 9. 备份与恢复

要求：

- 数据库定时备份
- 对象存储关键路径开启版本保护
- 审计日志不可轻易删除
- 恢复流程需定期演练

首期恢复目标建议：

- 结构化数据：可恢复到最近一次定时快照
- 关键项目产物：可恢复已选中版本
- 审计记录：默认长期保留
