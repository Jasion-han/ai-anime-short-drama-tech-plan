# AI 动漫短剧技术方案

更新日期：2026-03-18

## 1. 目标定义

本方案面向通过业界主流 AI 模型与生产工具协同完成一部 AI 动漫短剧的完整生产流程。文中提到的 `Claude / Gemini / GPT` 仅作为常见代表，不构成技术边界。实际执行时，应根据任务复杂度、模型能力特征、调用成本、上下文长度、响应时延与可获得性，在不同供应商的模型与工具之间进行调度，目标是形成一套可落地、可复用、成本可控、便于迭代返工的技术路线。

核心原则不是“把整部剧直接交给一个模型生成”，而是采用工程化流水线：

```text
剧本结构化 -> 角色/场景资产锁定 -> 分镜级镜头生产 -> 配音/音效 -> 后期剪辑合成 -> 质检返工
```

适用范围：

- 个人创作者
- 2 到 5 人的小团队
- 短视频平台竖屏短剧
- 横屏叙事短片
- 以二次元、日漫、anime look 为主的风格化内容

---

## 2. 总体制作思路

AI 动漫短剧更适宜采用多模型分层协作，而不是单模型端到端生成。这里的“多模型”并非限定于少数固定厂商，而是一个可按任务调度的模型与工具组合。

建议将工具链拆成 5 层：

| 层级 | 作用 | 可选模型池 |
|---|---|---|
| 文本中台 | 拆剧本、写分镜、角色设定、统一版本、生成提示词 | Claude / Gemini / GPT / Qwen / DeepSeek / Kimi / 豆包 / 混元 / 通义 / 智谱 等 |
| 视觉设定 | 角色立绘、表情包、场景图、道具图、风格锁定 | GPT Image / Imagen / Flux 系 / Runway References / 即梦 / 可灵图片等 |
| 视频生成 | 将关键帧或参考图生成镜头 | Flow + Veo / Runway / Sora / 可灵 / 即梦视频 / Pika / Luma 等 |
| 语音音频 | 配音、旁白、口播、多语种版本 | ElevenLabs / Gemini TTS / 国内 TTS 平台 / 商业配音工具 |
| 后期装配 | 剪辑、字幕、混音、转场、导出 | DaVinci Resolve / Premiere / CapCut |

默认推荐的协作原则：

- 低复杂度、高重复任务优先交给成本效率更优的模型批处理
- 中复杂度任务优先使用长上下文且价格适中的模型
- 高歧义、高风险决策再升级到更强模型处理
- 多模态看图、审片、连续性质检优先使用原生多模态能力更强的模型
- 图像、视频、配音优先使用对应赛道中稳定性更高的专业工具，而不是由文本模型替代专用生成工具承担全部生产职责

### 2.1 模型调度原则

建议把文本与多模态任务按复杂度分成 3 层：

#### L1：低复杂度、低成本层

适合：

- 文本清洗
- 剧本格式整理
- 批量改写提示词
- 批量生成镜头描述初稿
- 标签归类
- 字幕润色
- 重复性强的 CSV 补全

执行原则：

- 优先使用单位成本较低、响应速度较快且上下文容量满足需求的模型
- 适合批处理的任务，不宜采用逐条交互方式处理
- 结果优先满足覆盖率要求，再由上层模型进行少量抽检

#### L2：中复杂度、主生产层

适合：

- 场次拆解
- 镜头规划
- 角色设定整理
- 风格设定整理
- 中等复杂度提示词生成
- 粗剪问题分析

执行原则：

- 这是整个项目的主力层，应优先考虑成本效率与稳定性的平衡
- 可在不同供应商模型之间选择，前提是其在长文本理解、结构化输出和格式遵循方面表现稳定

#### L3：高复杂度、关键决策层

适合：

- 重写关键剧情节点
- 处理情绪弧线和隐含动机
- 解决剧本逻辑冲突
- 处理复杂多轮风格统一
- 审核关键镜头是否成立
- 最终方案把关

执行原则：

- 仅在关键环节调用，避免在全流程中持续使用最高能力档模型
- 将前序阶段整理完成的结构化材料提供给该层模型，以减少无效 token 消耗

### 2.2 成本优先的调度逻辑

默认决策顺序建议如下：

```text
先判断任务是否确实需要最高能力档模型 -> 能否拆分 -> 能否批处理 -> 能否先由低成本模型产出初版 -> 仅在关键节点升级
```

对于文本任务，优先遵循：

```text
低成本模型批量生成初版 -> 中等能力模型完成结构化整理 -> 高能力模型仅处理关键修改与审核
```

对于视觉任务，优先遵循：

```text
先以低成本方案产出参考图和 animatic -> 锁定方向 -> 再对关键镜头采用高质量工具重制
```

后文将在各具体步骤中直接标注推荐模型，不再单独设置集中汇总章节，以避免选型信息与执行步骤脱节。

## 3. 关键生产原则

### 3.1 按镜头生产，不按整场生成

每个镜头建议控制在 `3-8 秒`：

- 对话镜头：`3-5 秒`
- 动作镜头：`4-8 秒`
- 过场镜头：`2-4 秒`

不要直接生成 20 秒以上的复杂段落，否则容易出现：

- 角色漂移
- 构图失控
- 动作错乱
- 口型和对白无法对齐
- 成本快速上升

### 3.2 动漫短剧优先采用“图生视频”

对于对话、情绪、特写镜头，推荐先产出：

- 角色参考图
- 表情图
- 场景图
- 构图参考图

再用视频模型做轻动作、镜头运动和表情变化。  
相较于直接采用纯文生视频，这一路径在动漫风格项目中通常更稳定。

### 3.3 先做 Animatic，再做 Final

建议先形成一版低成本粗剪：

- 用占位图
- 用低成本视频模型
- 用临时配音
- 用占位音效

这一阶段适合的模型 / 工具：

- 分镜顺序和粗剪说明：`Gemini 2.5 Flash` / `Claude Sonnet 4.6`
- 占位图：即梦AI图片 / `Nano Banana`
- 低成本视频初版：即梦AI视频 / `Runway Gen-4 Turbo` / `Luma Dream Machine`
- 临时配音：`Gemini 2.5 Flash TTS` / 国内 TTS
- 粗剪质检：`Gemini 2.5 Pro`

先验证：

- 节奏是否成立
- 镜头顺序是否顺
- 对白时长是否匹配
- 情绪推进是否有效

确认后，再升级关键镜头质量。

### 3.4 所有改动只维护一个主版本

无论采用多少 AI 工具，项目层面都应维护单一主版本文件：

- `master_script.md`

所有场次、镜头、提示词、配音、字幕都以它为唯一来源，否则后面会出现：

- 台词版本冲突
- 分镜与对白不一致
- 字幕时间轴失效
- 配音返工成本增加

## 4. 项目标准文件结构

建议建立如下目录：

```text
project/
  00_docs/
  01_script/
    master_script.md
    scene_list.csv
    shot_list.csv
  02_storyboard/
  03_characters/
    hero/
    heroine/
    villain/
  04_backgrounds/
  05_props/
  06_audio/
    voices/
    dialogue/
    sfx/
    music/
  07_video_shots/
    sc01/
    sc02/
  08_edit/
  09_exports/
  10_qc/
```

建议同时维护以下文档：

- `master_script.md`
- `scene_list.csv`
- `shot_list.csv`
- `character_bible.md`
- `style_bible.md`
- `prompt_library.md`
- `asset_provenance_log.csv`（字段定义见第 14 节）
- `delivery_checklist.md`

### 4.1 文件命名规范

建议从项目开始即统一命名规则，避免进入剪辑与返工阶段后出现素材错位。

建议格式：

```text
scene_id: sc01
shot_id: sc01_sh03
take_id: sc01_sh03_take_02
audio_line_id: sc01_sh03_l01
subtitle_id: sc01_sh03_sub_01
```

建议原则：

- `scene_id`、`shot_id`、`take_id` 全项目唯一
- 镜头编号一旦进入 animatic，不再随意改号
- 删除镜头时保留原编号，不建议整表重排
- 所有导出文件都应带上 `scene_id` 与 `shot_id`

### 4.2 参考资产与版本绑定

角色、场景、道具与风格参考图建议按版本管理，不建议以“最新那张图”口头传递。

建议维护：

- `char_ref_version`
- `bg_ref_version`
- `prop_ref_version`
- `style_version`

执行原则：

- 每次关键调整后生成新版本，例如 `hero_ref_v02`
- `shot_list.csv` 中的镜头应绑定所使用的参考资产版本
- 同一场戏如无必要，不跨版本混用角色与场景参考图
- 只有完成审查的新版本，才允许进入后续镜头生产

## 5. 前期策划阶段

### 5.1 开工前必须锁定的核心决策

1. 成片长度
2. 画幅规格
3. 视觉风格
4. 配音方案

参考：

- 短视频平台优先：`9:16`
- 横屏叙事优先：`16:9`
- 时长建议：
  - 高传播短内容：`45-90 秒`
  - 完整剧情短片：`2-5 分钟`

正式进入资产制作前，建议同步锁定以下执行参数：

- 项目帧率，例如 `24fps` 或 `25fps`
- 预算上限
- 首发平台
- 首发语言
- 是否需要多语言版本

### 5.2 剧本结构化拆解

使用合适的长上下文文本模型或成本效率较优的结构化模型将剧本拆解为：

- 场次
- 情绪节点
- 冲突点
- 视觉重点
- 镜头需求
- 对白时长预估

产出物建议如下。

推荐模型：

- 首轮低成本拆解：`DeepSeek-V3.2` / `Gemini 2.5 Flash-Lite` / `GPT-5.4-nano`
- 主力结构化整理：`Claude Sonnet 4.6` / `Gemini 2.5 Flash` / `GPT-5.4-mini`
- 关键桥段审查与重写：`Claude Opus 4.6` / `GPT-5.4` / `Gemini 2.5 Pro`

建议用法：

- 先由低成本模型完成剧本切段、编号与场次初分
- 再让主力模型生成 `scene_list` 和 `shot_list`
- 仅当剧情逻辑、情绪走向或对白质量存在明显问题时，才调用高能力模型重写关键段落

#### `scene_list.csv`

至少包含：

```csv
scene_id,location,time_of_day,summary,characters,dramatic_goal,emotion,estimated_duration_s
```

#### `shot_list.csv`

至少包含：

```csv
shot_id,scene_id,beat,duration_s,shot_type,camera_motion,characters,dialogue,emotion,bg_ref,char_ref,prompt,status,model,seed,take,final_path
```

实际执行时建议补充：

```csv
char_ref_version,bg_ref_version,style_version,audio_line_id,selected_take,qc_status
```

#### `character_bible.md`

每个角色建议写清：

- 姓名
- 年龄感
- 性格关键词
- 服装设定
- 发型与发色
- 眼型
- 体型
- 色彩主调
- 常用表情
- 声线特征
- 禁止偏离项

#### `style_bible.md`

至少包含：

- 目标风格
- 线条粗细
- 上色方式
- 光影逻辑
- 镜头语言
- 色板
- 背景复杂度
- 人物与场景融合关系
- 禁止元素

#### `prompt_library.md`

至少包含：

- 角色提示词模板（按角色分组，含关联参考图 ID 与版本号）
- 场景提示词模板（按场景分组，含光照、时段、构图约束）
- 风格锁定词组（固定正向风格词与 Negative Prompt）
- 动作类型提示词（对话镜头、动作镜头、过场镜头各一套）
- 每条记录建议包含：用途标签、版本号、关联镜头 ID、最后修改日期
- 已废弃提示词单独标记，不删除，供返工时参考

### 5.3 项目冻结点与变更规则

AI 短剧项目不怕单次生成失败，最怕在生产中持续改剧本、改设定、改镜头目标。

建议至少设置 4 个冻结点：

1. 剧本冻结：`master_script.md` 的剧情、台词、场次顺序确认后，不再随意改主叙事
2. 资产冻结：主角色、主场景、关键道具的参考资产确认后，不在镜头生产中临时换风格
3. Animatic 冻结：粗剪节奏确认后，不再频繁改镜头顺序与时长结构
4. 成片冻结：进入最终混音、字幕、调色后，只处理明确缺陷，不再回到大规模重做

建议的变更规则：

- 冻结点之后的改动，必须说明改动原因、影响范围和返工成本
- 如改动影响超过 `3` 个镜头，应先更新 `master_script.md` 与 `shot_list.csv`
- 如改动涉及角色外观、场景布局或关键道具，应先升版参考资产，再进入重做
- 关键剧情改动优先在 animatic 阶段完成，不建议在最终镜头批量生成后再改

## 6. 美术资产阶段

这一阶段不宜急于进入视频生产，应先锁定可复用资产。

### 6.1 角色资产

每个主角建议准备：

- `8-12` 张角色参考图
- `6-10` 组表情变化
- 至少 `3` 个镜头距离版本：
  - 全身
  - 半身
  - 近景

建议固定的维度：

- 发型
- 发色
- 服装层级
- 饰品
- 年龄感
- 身高比例
- 眼型
- 肤色
- 主色系

推荐模型：

- 角色设定文字整理：`Claude Sonnet 4.6` / `GPT-5.4-mini` / `Gemini 2.5 Flash`
- 角色立绘和表情页：`GPT Image 1` / `Imagen 4` / `Nano Banana Pro`
- 低成本角色初版：即梦AI图片 / `Nano Banana`
- 局部修正：`GPT Image 1` / `SeedEdit 3.0`

### 6.2 场景资产

每个主场景建议准备：

- `3-6` 张环境图
- 日景 / 夜景版本
- 空镜版本
- 带角色站位预构图版本

重点包括：

- 教室
- 屋顶
- 街道
- 卧室
- 医院
- 地铁
- 公园
- 幻想空间

推荐模型：

- 场景文案整理：`Gemini 2.5 Flash` / `Claude Sonnet 4.6`
- 场景参考图：`Imagen 4` / `GPT Image 1`
- 中文快速生成初版：即梦AI图片
- 已有底图时的视角、光影、背景修正：`SeedEdit 3.0` / `GPT Image 1`

### 6.3 道具资产

关键道具应独立出图，例如：

- 手机
- 项链
- 武器
- 书本
- 雨伞
- 校徽
- 信件

原因是关键道具在镜头里常常承担叙事信息，不能每次随机生成。

推荐模型：

- 道具描述整理：`DeepSeek-V3.2` / `Gemini 2.5 Flash-Lite`
- 高一致性道具出图：`GPT Image 1` / `Imagen 4`
- 低成本道具初版：即梦AI图片 / `Nano Banana`
- 细节修正：`SeedEdit 3.0` / `GPT Image 1`

## 7. 分镜与镜头生产方案

推荐把镜头分成 3 类处理。

### 7.1 对话镜头

适合：

- 站桩对话
- 情绪戏
- 心理戏
- 特写戏

推荐流程：

```text
角色/场景参考图 -> 构图图 -> 轻动作视频 -> 单独配音 -> 口型同步 -> 合成
```

推荐模型：

- 构图与镜头说明：`Claude Sonnet 4.6` / `Gemini 2.5 Flash`
- 参考图：`GPT Image 1` / `Imagen 4`
- 轻动作视频：`Runway Gen-4` / `Veo 3.1`
- 低成本视频初版：即梦AI视频 / `Runway Gen-4 Turbo`
- 粗剪配音：`Gemini 2.5 Flash TTS`
- 定稿配音：`ElevenLabs` / `Gemini 2.5 Pro TTS`

优势：

- 人物稳定
- 表情更可控
- 返工更低
- 更适合二次元风格

### 7.2 动作镜头

适合：

- 奔跑
- 追逐
- 转身
- 打斗
- 爆发
- 跳跃
- 镜头绕拍

建议：

- 每个动作拆成短镜头
- 不宜一次性生成多角色复杂交互
- 高动作戏建议优先制作参考帧
- 多做几个 take 再筛选

推荐模型：

- 动作镜头规划：`Claude Opus 4.6` / `Gemini 2.5 Pro` / `GPT-5.4`
- 初版验证：即梦AI视频 / `Runway Gen-4 Turbo` / `Luma Dream Machine`
- 定稿生成：`Veo 3.1` / `Runway Gen-4` / `Sora 2`

使用原则：

- 动作戏不建议直接使用最高成本模型
- 应先以低成本工具验证镜头节奏、时长与动作拆分是否成立
- 通过后再升级到高质量视频模型

### 7.3 过场镜头

这是提升完成度的重要手段，用于连接、遮丑、补节奏。

常见类型：

- 天空
- 城市
- 校园走廊
- 风吹窗帘
- 手部特写
- 门把手
- 手机屏幕
- 鞋子停下
- 角色回头

很多 AI 动漫片的流畅感，其实依赖的是过场镜头安排得足够聪明。

推荐模型：

- 过场文案生成：`DeepSeek-V3.2` / `Gemini 2.5 Flash-Lite`
- 过场视频生成：即梦AI视频 / `Runway Gen-4 Turbo` / `Luma Dream Machine`

使用原则：

- 过场镜头优先采用低成本、高吞吐的批量生成方案
- 这类镜头通常无须使用最高能力档视频模型

### 7.4 镜头级验收标准

每个镜头在进入下一阶段前，建议至少检查以下项目：

1. 角色一致性：脸型、发型、服装、饰品是否与参考资产一致
2. 画面可读性：构图是否清楚，主体是否明确，关键信息是否被遮挡
3. 动作与情绪成立：动作是否可理解，表情与台词情绪是否匹配
4. 时长匹配：镜头长度是否与对白、动作和剪辑节奏匹配
5. 衔接可用性：是否能顺利接入前后镜头，是否需要补过场或反应镜头

建议把镜头状态至少分为：

- `draft`
- `review`
- `approved_for_edit`
- `redo`

只有进入 `approved_for_edit` 的镜头，才建议投入高成本精修或正式剪辑。

### 7.5 失败回退与重做策略

同一镜头不建议无限次重跑，应设置明确的回退顺序。

建议顺序：

1. 先检查参考图是否稳定，避免在错误资产上重复生成
2. 再调整镜头描述、构图约束与动作拆分
3. 连续多次失败时，优先拆镜头而不是继续增加提示词复杂度
4. 如动作仍不稳定，可改为静帧推拉、反应镜头或过场镜头替代
5. 只有在镜头本身成立但质量不足时，才升级到更高成本工具重做

执行建议：

- 单镜头连续 `3-5` 次失败后必须暂停，先复盘失败原因
- 不同工具的失败样本要保留，避免重复试错
- 回退动作优先级应高于继续堆 token 或盲目切换模型

## 8. AI 协作原则

本节不再单独列出模型名单，相关推荐已分别写入前后对应步骤。

协作时只保留 4 个原则：

1. 低复杂度文本任务优先由成本效率更优的模型批量处理
2. 主策划与结构化整理交给长上下文且稳定的模型
3. 图像、视频、配音优先使用对应赛道最稳定的专业工具
4. 多模态模型主要负责审图、审片、连续性检查，不直接接管主剧本

## 9. 模型与视频工具调度策略

### 9.1 选型原则

选型时不应先以厂商名称为起点，而应先回答以下问题：

1. 这是文本任务、图像任务、视频任务，还是多模态审查任务？
2. 这是低复杂度重复任务，还是高风险关键任务？
3. 这个任务更在意质量、速度，还是成本？
4. 这个任务能否拆分后交由更低成本的模型处理？

默认原则：

- 文本先分层调度
- 图像优先稳定参考资产
- 视频优先镜头级生成
- 配音优先独立音轨
- 多模态模型主要做审查，不做主生产

### 9.2 三种默认调度方案

#### 方案 A：质量优先

- 文本：使用对应步骤中的高质量选项
- 图像：使用高一致性定稿方案
- 视频：使用高质量关键镜头方案
- 配音：使用定稿级配音方案或真人配音
- 后期：`DaVinci Resolve`

适合：

- 追求镜头质感
- 需要更强电影感
- 有一定预算

#### 方案 B：成本优先

- 文本：使用对应步骤中的低成本文本方案
- 图像：优先采用初版出图方案
- 视频：优先采用初版视频方案
- 配音：优先采用粗剪阶段配音方案
- 后期：`CapCut / Resolve`

适合：

- 快速验证方案
- 先完成最小可行版本
- 预算有限

#### 方案 C：混合优先

- 文本：批量任务与关键决策分开选型
- 图像：初版与定稿分开选型
- 视频：普通镜头与关键镜头分开选型
- 配音：粗剪与定稿分开选型
- 后期：按团队熟悉度选 `Resolve / Premiere / CapCut`

适合：

- 想兼顾质量和成本
- 需要利用不同模型各自优势
- 不希望被单一平台绑定

### 9.3 视频工具的实际用法

建议把视频工具分为两类：

- 高质量关键镜头工具
- 低成本初版与试错工具

具体执行时：

- 对话镜头优先走图生视频
- 高动作镜头优先短拆并多 take
- 过场镜头优先采用低成本工具进行批量生成
- 只有入选镜头才进入高成本重做

具体模型选择已在 `7.1 对话镜头`、`7.2 动作镜头`、`7.3 过场镜头` 中就地标注。

补充原则：

- 需要保留已有运动基础时，优先考虑支持 `Modify / Reframe` 一类工作流的视频工具

### 9.4 镜头生成参数建议

每个镜头记录这些参数：

- 模型
- 版本
- Prompt
- Negative Prompt
- Seed
- Duration
- Aspect Ratio
- 参考图 ID
- Take 编号
- 是否入选

建议对每个镜头至少保留：

- `take_01`
- `take_02`
- `selected_take`

## 10. 声音与配音方案

声音质量直接决定成片完成度。

### 10.1 推荐的音频流程

```text
角色声线设计 -> 逐句对白生成/录制 -> 口型同步 -> SFX -> BGM -> 混音
```

推荐模型：

- 粗剪占位配音：`Gemini 2.5 Flash TTS` / 国内 TTS 平台
- 成片配音：`ElevenLabs` / `Gemini 2.5 Pro TTS`
- 多语言扩展：优先 `ElevenLabs` 或具备多语种稳定输出的商业 TTS
- AI 音乐初版：`Lyria Experimental`

### 10.2 BGM 使用原则

BGM 来源建议按以下优先级选择：

1. 授权版权曲库（如 Epidemic Sound / Artlist）：商用风险最低，适合正式发布
2. AI 生成音乐（如 `Lyria Experimental`）：适合粗剪占位与风格验证，正式商用前需确认平台条款
3. 自制或委托创作：成本较高，适合有明确品牌需求的项目

执行建议：

- BGM 音量应低于对白，保证对白优先级
- 粗剪阶段可使用 AI 生成占位音乐，定稿前替换为授权曲目
- BGM 曲目来源与授权范围应记录至 `asset_provenance_log.csv`

### 10.3 音频工程规范

建议统一：

- 采样率：`48kHz`
- 主项目音频格式：`WAV`
- 每句对白单独编号
- 字幕时间轴与对白编号一一对应
- 建议统一响度标准与峰值上限
- 每句对白前后预留少量静音，便于口型与剪辑微调

### 10.4 为什么不要依赖视频模型的最终对白

原因：

- 改词困难
- 情绪不稳定
- 语速不可控
- 多语言版本复用差
- 后期精修空间有限

因此建议：

- 最终对白统一走独立音轨
- 口型作为后续匹配工序
- 便于多版本导出

## 11. 后期剪辑与合成

推荐使用：

- `DaVinci Resolve`
- `Premiere Pro`
- `CapCut`

### 11.1 后期流程建议

```text
导入镜头 -> 按 shot_id 排列 -> 铺对白 -> 对齐动作 -> 补空镜 -> 上字幕 -> 加 SFX -> 铺 BGM -> 调色统一 -> 导出
```

### 11.2 后期重点

- 统一镜头节奏
- 用空镜和反应镜头遮蔽镜头接缝
- 统一调色减少模型差异
- 字幕统一字重、描边、位置
- 保证对白优先级高于音乐

### 11.3 推荐导出版本

- `Master 1080p`
- `9:16 发布版`
- `16:9 横版`
- `Clean 无字幕版`
- `Caption 带字幕版`

### 11.4 技术交付规格

为避免后期阶段因工程设置不一致产生返工，建议在项目开始时统一以下规格：

- 时间线分辨率
- 项目帧率
- 工作色彩空间
- 字幕安全区
- 竖屏标题安全区
- 字体授权状态
- 导出编码规范

建议原则：

- 素材、时间线与最终导出尽量保持同一帧率
- 平台发布版与母版分开导出，不建议共用单一压缩文件
- 字幕、标题、贴纸等叠加元素应提前按平台安全区检查
- 所有交付版本应与 `delivery_checklist.md` 对照验收

#### `delivery_checklist.md`

建议至少包含以下核查项：

- [ ] 所有镜头状态均为 `approved_for_edit`
- [ ] `shot_list.csv` 与实际剪辑时间线一一对应
- [ ] 字幕时间轴与对白编号核对完毕
- [ ] 响度标准与峰值上限符合目标平台要求
- [ ] 所有导出版本（竖屏、横屏、无字幕、带字幕）已生成
- [ ] 字幕、标题位置在平台安全区内
- [ ] `asset_provenance_log.csv` 已更新至最终使用资产
- [ ] 平台要求的 AI 生成披露标记已添加
- [ ] 供应商商用条款核对完毕
- [ ] 母版文件与发布压缩版分开存档

## 12. 成本控制方案

AI 动漫项目中，预算大头通常是视频生成，不是 LLM 文本。

### 12.1 预算结构参考

- 文本规划：小于总预算的 `5%`
- 图像资产：约 `10%-20%`
- 视频生成：约 `60%-80%`
- 配音与后期：约 `10%-20%`

### 12.2 粗估方法

例如一部 `3 分钟` 短剧：

- 假设 `60` 个镜头
- 平均每个镜头 `5` 秒
- 总成片约 `300` 秒
- 每个镜头先跑 `2` 个 take

则总生成量约：

```text
60 x 5 x 2 = 600 秒生成素材
```

### 12.3 成本优化原则

- 粗剪阶段优先使用低成本模型
- 关键镜头再升级至高能力模型
- 不建议全片持续使用最高能力档模型
- 先验证对白和节奏，再升级画面
- 多做静帧和过场，减少高成本动作镜头占比
- 低复杂度文本任务统一采用成本效率更优的模型批处理
- 长剧本宜先拆段、摘要、合并，避免反复将全文输入高成本模型
- 尽量使用固定模板和结构化字段，减少重复提示词 token
- 建议先由低成本模型完成可用初版，再由高能力模型完成关键修正

具体到模型的建议：

- 文本、图像、视频、配音的具体模型，统一以各步骤中第一次出现的标注为准
- 低成本方案只用于初稿、粗剪、验证节奏
- 高质量方案只用于关键镜头、关键对白、最终定稿

## 13. 最容易翻车的 10 个问题

1. 角色脸漂
   解决：使用固定 reference pack，统一角色设定图（见第 6.1 节）

2. 服装漂移
   解决：服装写成硬约束，不要模糊描述（见第 5.2 节 `character_bible.md`）

3. 场景不一致
   解决：主场景先建背景资产，不临时随机生成（见第 6.2 节）

4. 对白节奏不对
   解决：先做音频 animatic，再做镜头（见第 3.3 节）

5. 口型不自然
   解决：最终口型单独处理，不依赖随机开口（见第 10.4 节）

6. 打斗画面崩坏
   解决：动作戏拆短镜头，降低复杂交互（见第 7.2 节）

7. 成本失控
   解决：先以低成本方案验证，再对关键镜头升级（见第 12 节）

8. 剧情节奏拖沓
   解决：短剧保持高信息密度，减少冗长停顿（见第 3.1 节）

9. 整片风格不统一
   解决：提前建立 `style_bible.md` 和 `prompt_library.md`（见第 5.2 节）

10. 法务风险
    解决：避免直接模仿已有知名 IP、声线、品牌或角色形象（见第 14 节）

## 14. 合规与版权要求

正式发布前，建议重点检查：

- 是否直接模仿某知名动漫 IP 的角色设计
- 是否未经授权使用真人声音克隆
- 是否使用未经授权的人像作为角色参考
- 是否保留素材来源与模型版本记录
- 是否包含平台要求披露的 AI 生成标记

建议维护：

- `asset_provenance_log.csv`

至少记录：

- 资产 ID
- 生成日期
- 工具名称
- 模型版本
- Prompt
- Seed
- 参考图来源
- 使用范围
- 是否商用

### 14.1 供应商条款与数据留存检查

除版权来源外，还应逐项核对所使用平台的服务条款与数据策略。

重点检查：

- 是否允许商用
- 是否允许二次编辑与再发布
- 是否限制声音克隆、人像输入或品牌元素生成
- 上传素材是否会被平台留存
- 是否支持关闭训练留存或零数据留存选项
- 团队内部未公开剧本是否需要先做脱敏或拆段处理

执行建议：

- 关键剧本、未发布角色设定与商业合作素材，不建议无差别上传到所有外部平台
- 同一项目尽量建立已批准工具清单，避免成员各自选择条款不一致的平台
- 如某平台条款存在歧义，宁可将其限制在初稿、粗剪或非商用试验阶段

## 15. 标准执行流程

建议按如下顺序启动项目：

1. 将剧本整理成纯文本版
2. 生成 `scene list`
3. 生成 `shot list`
4. 建立 `character bible`
5. 建立 `style bible`
6. 先做角色和场景资产
7. 生成低成本 animatic
8. 锁节奏后重做关键镜头
9. 统一配音、音效、字幕
10. 后期剪辑、混音、导出

以上每一步所适合的模型，均已在对应章节中就地标注，不在这里重复展开。

## 16. 推荐的实际落地打法

如果你想以最低风险开始，建议采用下面这套默认打法：

### 阶段一：策划

- 先由成本效率更优的文本模型完成剧本清洗、分场和结构化初版
- 再用更强的长上下文模型审查剧情、情绪和镜头逻辑
- 产出 `scene_list.csv` 和 `shot_list.csv`
- 人工确认节奏和时长

### 阶段二：美术锁定

- 用合适的图像模型生成角色设定图和主场景参考图
- 固定表情、服装、发型、颜色
- 制作主场景背景图

### 阶段三：粗剪

- 使用低成本视频模型生成镜头初版和过场镜头
- 临时配音
- 形成第一版 animatic

### 阶段四：精修

- 升级关键镜头到更高质量的视频工具
- 重做动作镜头
- 统一口型和声音

### 阶段五：成片

- 加字幕
- 加音效
- 混音
- 调色
- 导出多个平台版本
